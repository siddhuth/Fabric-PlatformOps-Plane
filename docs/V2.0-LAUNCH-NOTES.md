# Fabric Access Platform — V2.0 Launch Notes

## Multi-Platform Access Control Plane: Fabric + Databricks + Snowflake (Future)

---

## Overview

Version 2.0 extends the Fabric Access Platform from a single-platform Fabric permission manager into a **multi-platform access control plane** spanning Microsoft Fabric, Azure Databricks (with Unity Catalog), and future Snowflake integration.

The release is staged across six sequential feature branches, each independently reviewable and mergeable. The end state delivers:

- **Multi-platform YAML access packages** — persona-based permission bundles that span Fabric and Databricks in a single definition
- **Unified provisioning engine** — provider-abstracted Azure Functions that orchestrate grants across both platforms from a single Entra group trigger
- **Working demo experience** — React dashboard with synthetic data, animated provisioning flows, cross-platform access matrix, audit timeline, and drift detection
- **Infrastructure-as-code** — Terraform modules for Databricks workspace, Unity Catalog, and SCIM identity sync
- **CI/CD pipelines** — GitHub Actions for schema validation, fixture consistency, UI builds, and demo deployment
- **Snowflake extensibility seams** — schema support, provider stubs, and UI placeholders ready for Phase 3

## Databricks Permission Model — Key Design Decisions

### Unity Catalog: Positive-Permission-Only

Unlike Fabric SQL endpoints which support `GRANT`/`DENY`/`REVOKE`, Unity Catalog is **positive-permission-only** — there is no `DENY`. The V2.0 YAML schema reflects this:

```yaml
platforms:
  databricks:
    unity_catalog:
      grants: [...]        # What to GRANT
      # No deny[] array — UC doesn't support it
```

### Two Parallel Permission Systems

| System | Governs | Mechanism |
|--------|---------|-----------|
| **Unity Catalog** (data plane) | Catalogs, schemas, tables, views, volumes, functions, models, storage credentials, external locations, connections | SQL `GRANT`/`REVOKE` |
| **Workspace ACLs** (control plane) | Clusters, SQL warehouses, jobs, DLT pipelines, notebooks, repos, serving endpoints, Genie spaces, dashboards, apps | REST API `PUT /api/2.0/permissions/{type}/{id}` |

Both must be provisioned — a user needs workspace ACLs on compute **and** UC grants on data.

### Prerequisite Chain Enforcement

UC requires `USE CATALOG` + `USE SCHEMA` before any object-level privilege takes effect. The validation script enforces this:

```
✗ ERROR [databricks-engineer.yaml]: UC grant SELECT on sales_catalog.staging.orders
  requires USE SCHEMA on sales_catalog.staging (not declared)
```

### Identity: Same Entra Groups, Dual-Platform

```
Entra ID ──(SCIM)──► Databricks Account ──► Workspace(s)
    │                                          │
    └──(existing)──► Fabric Workspace          │
                                               │
    sg-fabric-de-sales ─── same group ─── both platforms
```

Only **account-level groups** (synced via SCIM) can be used as UC grant principals.

### Supported Databricks Item Types

**Unity Catalog Securables:**

| Type | Key Privileges |
|------|---------------|
| Catalog | USE CATALOG, CREATE SCHEMA, BROWSE |
| Schema | USE SCHEMA, CREATE TABLE/VIEW/VOLUME/FUNCTION/MODEL, SELECT, MODIFY |
| Table | SELECT, MODIFY |
| View | SELECT |
| MaterializedView | SELECT |
| Volume | READ VOLUME, WRITE VOLUME |
| Function | EXECUTE |
| RegisteredModel | APPLY MODEL |
| ExternalLocation | READ FILES, WRITE FILES, CREATE EXTERNAL TABLE |
| StorageCredential | CREATE EXTERNAL LOCATION, READ FILES, WRITE FILES |
| Connection | USE CONNECTION, CREATE FOREIGN CATALOG |

**Workspace ACL Objects:**

| Type | Permission Levels |
|------|------------------|
| Cluster | CAN_ATTACH_TO, CAN_RESTART, CAN_MANAGE |
| ClusterPolicy | CAN_USE, CAN_MANAGE |
| SQLWarehouse | CAN_USE, CAN_MONITOR, CAN_MANAGE |
| Job | CAN_VIEW, CAN_MANAGE_RUN, CAN_MANAGE |
| DLTPipeline | CAN_VIEW, CAN_RUN, CAN_MANAGE |
| Notebook | CAN_READ, CAN_RUN, CAN_EDIT, CAN_MANAGE |
| Repo | CAN_READ, CAN_RUN, CAN_EDIT, CAN_MANAGE |
| ServingEndpoint | CAN_VIEW, CAN_QUERY, CAN_MANAGE |
| GenieSpace | CAN_VIEW, CAN_RUN, CAN_EDIT, CAN_MANAGE |
| AIBIDashboard | CAN_VIEW, CAN_RUN, CAN_EDIT, CAN_MANAGE |
| DatabricksApp | CAN_VIEW, CAN_USE, CAN_MANAGE |

**Entitlements (SCIM):** workspace-access, databricks-sql-access, allow-cluster-create, allow-instance-pool-create

### System Tables for Drift Detection

| System Table | Use |
|---|---|
| `system.information_schema.table_privileges` | Compare declared grants vs actual |
| `system.information_schema.schema_privileges` | Schema-level grant verification |
| `system.information_schema.catalog_privileges` | Catalog-level grant verification |
| `system.access.audit` | Audit trail of GRANT/REVOKE events |
| `system.access.table_lineage` | Data flow tracking |
| `system.billing.usage` | Cost attribution per persona |
| `system.compute.clusters` | Cluster policy compliance |

## Snowflake Parity Map (Phase 3 Reference)

| Framework Layer | Fabric | Databricks UC | Snowflake (future) |
|----------------|--------|---------------|-------------------|
| Identity source | Entra group | Same Entra group (SCIM) | Same Entra group (SCIM) |
| Workspace access | Workspace role | Entitlements + assignment | Account role + warehouse |
| Data grants | PBI API item sharing | SQL `GRANT` on UC securables | SQL `GRANT` on database objects |
| Compute access | N/A (serverless) | Cluster policy + SQL warehouse ACL | Warehouse `USAGE` grant |
| Drift detection | PBI Admin API | `system.information_schema.*_privileges` | `INFORMATION_SCHEMA` / `SHOW GRANTS` |
| Audit | Log Analytics | `system.access.audit` | `SNOWFLAKE.ACCOUNT_USAGE.ACCESS_HISTORY` |

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Schema change breaks existing Fabric packages | `platforms` key is optional; existing packages without it remain valid |
| Provider refactoring breaks live Fabric functions | Existing `fabric_client` internals untouched; new interface wraps existing methods. `DEMO_MODE` is opt-in. |
| Fixture data drifts from YAML packages | Generator reads actual YAMLs. CI workflow auto-regenerates and commits on PR. |
| UI scope creep | Strictly fixture-driven; no live API integration needed for demo. |
| Terraform modules conflict with existing | Purely additive in new directories; no modification to existing modules. |
